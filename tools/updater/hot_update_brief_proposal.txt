SanityPrint Windows 热更新方案预研（简版）
====================================

一、方案概述
------------
1. 建议方案
   - 继续使用 Squirrel.Windows 作为“打包与增量格式”的生产工具：
     - CI 中生成 RELEASES、full.nupkg、delta.nupkg；
     - 不使用 Squirrel 自带 Update.exe 做升级。
   - 自研一个独立的 Go 版本 updater.exe 作为“更新执行引擎”：
     - 负责在本地执行 delta 合成、目录切换和回滚；
     - 对用户只暴露一个简单的更新提示框。

2. 核心思路（一句话）
   - “线上仍然发 Squirrel 格式的 full + delta 包，但更新过程完全由我们自己的 updater 接管”。

3. 关键能力
   - 支持增量更新：在本地将当前安装目录视为 FULL 基线，叠加一个或多个 delta 包生成新版本；
   - 原子切换与回滚：安装目录先整体备份，再整体替换；失败时自动回滚；
   - 简单可控的更新 UI 和完整日志。


二、方案可行性分析
------------------
1. 技术成熟度
   - Squirrel.Windows 已在 Electron 等生态广泛使用，包格式和 delta 机制成熟；
   - Go 语言跨平台单文件 exe，已在本地验证可稳定完成解包、差分和文件操作；
   - 当前 PoC 已完成从 6.0.0-alpha → 6.0.1-alpha（full）、6.0.1-alpha → 7.0.0-alpha（delta）的完整升级链验证。

2. 与现有架构的契合度
   - 打包侧：
     - 已经在 CMake/Jenkins 中集成 Squirrel 打包脚本，无需大改现有流水线；
     - 版本号、渠道（测试/正式）等信息可以沿用现有配置。
   - 客户端侧：
     - updater.exe 独立于主程序，只依赖 Windows 系统库；
     - 主程序只需在合适时机退出自身并调用 updater.exe，不强绑定任何语言/框架。

3. 风险控制能力
   - 回滚保障：
     - 每次更新前，将现有安装目录整体重命名为带时间戳的备份目录；
     - 只有新目录完整构建成功并通过关键文件校验后，才进行替换；
     - 替换失败时，尝试使用备份目录恢复。
   - 校验机制：
     - 对 delta 包中的差分文件做大小和哈希校验，异常时自动降级为使用 full 包或保留旧文件；
     - 更新过程中的每一步操作都有日志记录，便于追踪。

4. 预研结果小结
   - 从“功能完整性”和“稳定性”角度，当前方案可行；
   - 实现复杂度适中，主要新增的是一个 Go 工具和少量 CI 脚本；
   - 未来如需回退到“只发 full 包”或引入其他更新框架（如自定义 HTTP 分发），Squirrel 产物仍然可复用。


三、主要隐患与边界
------------------
1. 依赖 Squirrel 生态
   - 隐患：
     - 包结构和 delta 机制依赖 Squirrel；若后续 Squirrel 停更或做重大变更，需要跟进适配；
     - 个别系统级 DLL 的 delta 质量可能不稳定。
   - 缓解：
     - 我们的 updater 只依赖“当前已验证”的包结构，版本升级可控；
     - 对单个 DLL 的 delta 失败采用容错处理（跳过或回退到 full），不会拖垮整次更新；
     - 保留 full 包作为随时可用的兜底方案。

2. 本地环境复杂性
   - 隐患：
     - 用户机器上可能存在杀毒软件、同步盘等，对文件锁定，导致安装目录 rename 或复制失败；
     - 磁盘空间不足可能在解包或备份阶段才暴露。
   - 缓解：
     - updater 已实现“rename 优先、复制兜底”的双路径切换逻辑，并在失败时尝试回滚；
     - 可以在更新前增加简要空间检查；更新失败时通过日志和提示文案引导用户清理空间或关闭相关程序。

3. 体验层面
   - 隐患：
     - 当前 PoC 的 UI 较为简单，只有一个“正在更新”的模态提示框，没有细颗粒度进度；
     - 对于大版本更新或网络较慢环境，用户可能感觉“卡住”。
   - 缓解：
     - 后续可以在不改动核心逻辑的前提下，增加进度上报机制（例如阶段性文案、粗粒度百分比）；
     - 在主程序中展示“更新日志/版本说明”，让用户对更新过程和收益有预期。

4. 边界条件
   - 方案当前聚焦于：
     - Windows 平台；
     - 单实例安装（一个安装目录）。
   - 不在本阶段覆盖的内容：
     - 跨平台统一更新方案；
     - 多安装路径、多账号共享安装等复杂场景。


四、结论与推进建议
------------------
1. 结论
   - 在不大改现有发布和客户端架构的前提下，使用“Squirrel 打包 + 自研 Go Updater 执行”的组合，可以较低成本为 SanityPrint 引入稳定的热更新能力；
   - 从当前验证结果来看，方案在技术上可行、风险可控，能显著改善用户升级体验和线上发版效率。

2. 建议的推进路径
   - 第一阶段（内部试点）：
     - 在 Alpha/内测渠道上线该更新机制，观察稳定性和用户反馈；
     - 完善日志、监控和回滚验证流程。
   - 第二阶段（对外推广）：
     - 在正式版中逐步替换原有“手工下载安装”的流程，默认启用热更新；
     - 配合运营与产品，设计更友好的更新提示和版本说明。
   - 第三阶段（能力增强）：
     - 增加灰度发布、强制升级、签名校验等能力，向“统一升级平台”演进。


五、更新流程设计（下载逻辑 + 本地 Updater）
-----------------------------------------
1. 更新检查与下载（由主程序完成）
  - 更新检查接口：
    - 主程序在合适时机调用更新服务接口（例如 `POST /api/update/check`），通过 JSON 体携带当前版本号、渠道、平台等参数；
     - 服务器返回 JSON，结构示例：
       - `hasUpdate`：是否有可用新版本；
      - `latestVersion`：目标版本号（例如 `7.0.0-alpha`）；
      - `basePackage`：当前安装版本对应的 full 包文件名（例如 `SanityPrint-6.0.1-alpha-full.nupkg`，用于本地 `RELEASES` 第一行）；
       - `fullFile`：full 包信息（`name`、`url`、`size`、`sha1`、`packageType` 等）；
       - `deltaFiles[]`：可用 delta 包列表（每个元素包含 `name`、`url`、`size`、`sha1` 等）；
       - 其他字段：渠道、更新说明、是否强制更新等。
   - 策略决策（delta / full）：
     - 若 `hasUpdate = false`：直接结束，本轮不更新；
     - 若 `hasUpdate = true`：
      - 服务端会基于 RELEASES 中的 full/delta 信息，预先计算从 `currentVersion` 到 `latestVersion` 的升级链，并返回 target 的 full 包（`fullFile`）以及从 `currentVersion` 到 `latestVersion` 过程中涉及到的所有 delta 包（`deltaFiles`），**不再在服务端做“delta vs full”大小比较和策略筛选**；
      - 对主程序而言，只需遵循简单约定：
         - 若 `deltaFiles` 非空：视为“delta 更新”，需要下载 `deltaFiles` 列表中的所有增量包；
         - 若 `deltaFiles` 为空：视为“full 更新”，只下载 `fullFile` 对应的 full.nupkg；
       - 可选：在某些场景下，可以同时下载 full 包作为兜底，以便在 delta 出问题时人工或后续版本使用。
   - 文件下载与校验：
     - 主程序在本地选择一个更新工作目录（当前实现使用 `%LOCALAPPDATA%\sanityprint_squirrel`），后续所有更新文件都放在该目录下；
     - 按前一步确定的策略，只下载本次需要的 full.nupkg 或若干 delta.nupkg（不再从服务器下载 `RELEASES` 文件）；
     - 下载完成后，对每个 `.nupkg` 执行基础校验：
       - 检查实际大小是否等于服务器返回的 `size`；
       - 计算整体 SHA1 是否等于服务器返回的 `sha1`；
       - 任意文件校验失败则放弃本轮更新，保留现有版本。
   - 本地更新计划落地：
     - 主程序在下载完成并校验通过后，根据 JSON 中的 fullFile/deltaFiles 列表，在更新目录中生成一个简化版 `RELEASES` 文件（每个包一行，包含 sha1 / 文件名 / size 等字段）；
     - 然后将本次更新所需的信息在本地“落地”为一个更新计划（可以是配置文件，也可以通过命令行参数传递）：
       - 安装目录 `installDir`；
       - 当前版本号 `currentVersion`；
       - 目标版本号 `targetVersion`；
       - 本次下载的 `.nupkg` 文件列表及所在目录；
     - 然后关闭自身主进程，启动 `updater.exe`，并传入：
       - `--install-dir=<安装目录>`；
       - `--current-version=<当前版本>`；
       - 工作目录指向刚才的临时更新目录（其中含有 `RELEASES` 与本次下载的所有 `.nupkg`）。

2. 本地更新执行（updater.exe）
   - 启动与准备：
     - `updater.exe` 启动后首先读取命令行参数，确认：
       - 当前安装目录 `installDir`；
       - 当前版本号 `currentVersion`；
       - 自身所在目录作为 `baseDir`（即刚才的临时更新目录）；
     - 在独立 goroutine 中弹出一个“正在更新”的系统模态提示框，避免用户误操作；
     - 打开 `baseDir` 下的 `RELEASES` 文件，解析出当前可用的版本列表及对应的 full/delta 包文件名。
   - 规划升级链（基于 RELEASES）：
     - 结合 `currentVersion` 和 `RELEASES` 中的版本信息，按版本号从低到高计算“升级链”，例如：
       - 当前为 `6.0.1-alpha`，`RELEASES` 中存在 `7.0.0-alpha` 的 full/delta 记录；
       - 则升级链形如：`[7.0.0-alpha]`；
     - 这些版本对应的 full/delta nupkg 文件，正是之前下载阶段已经放入 `baseDir` 的那一批。
   - 构建目标版本文件树：
     - 调用内部的 `buildTargetTree` 逻辑，在 `baseDir/process_full/<targetVersion>` 下构建出一个完整的目标版本文件树，核心步骤包括：
       - 选择基线：
         - 若升级链的首个版本存在 full 包，则从该 full.nupkg 解包到目标目录，作为基线；
         - 否则，将当前安装目录 `installDir` 视为 FULL 基线，整体拷贝到目标目录；
       - 依次应用升级链中的各个版本：
         - 若该版本存在 delta 包：
           - 调用 `applyDeltaNupkg`，对基线目录中的 EXE/DLL 做增量合成；
           - 对每个带 `.shasum` 的文件，按照 Squirrel 规则进行大小 + SHA1 校验，校验通过才替换旧文件；
           - 单个文件 delta 失败时，会记录日志并保留旧文件，不会写入损坏文件；
         - 若该版本只存在 full 包：
           - 直接用对应 full.nupkg 解包覆盖目标目录；
     - 若构建目标版本文件树过程中出现不可恢复错误（如 full 包缺失、解包失败），updater 会中断更新，写日志并退出。
   - 拷贝到新版本目录：
     - 目标版本文件树生成成功后，updater 在安装目录的上级目录下创建一个临时的新版本目录 `newDir`；
     - 将目标文件树整体拷贝到 `newDir` 中；
     - 检查 `newDir` 根目录下是否存在关键可执行文件 `SanityPrint.exe`：
       - 若不存在，则认为更新构建失败，删除 `newDir` 并退出。
   - 切换目录与回滚：
     - 若上述检查通过，则执行“原子切换”流程：
       - 将当前安装目录 `installDir` 重命名为带时间戳的备份目录 `backupDir`；
       - 尝试将 `newDir` 重命名为 `installDir`；
       - 若重命名失败（例如跨盘或被占用）：
         - 创建新的 `installDir`；
         - 将 `newDir` 中的文件通过逐文件复制的方式迁移过去；
         - 若复制也失败，则尽力将 `backupDir` 还原回原路径，保留旧版本；
     - 切换成功后：
       - 记录“从 currentVersion → targetVersion 更新成功”的日志；
       - 删除 `backupDir`（不再保留旧版本备份，以避免磁盘占用持续增长）；
       - 关闭“正在更新”提示框，正常退出。
     - 任意阶段出现致命错误时：
       - 保证不覆盖原有安装目录，或在失败后尽最大努力恢复 `backupDir`；
       - 将错误原因（失败文件、异常堆栈等）写入日志，方便后续排查。

3. 关键点小结
   - 更新策略前置：
     - “用 delta 还是 full”在下载阶段已经决定，updater.exe 只负责在本地按既有的 full/delta 包构建目标版本，不再自行做复杂策略判断；
   - 文件级校验：
     - 下载阶段对 `.nupkg` 包整体做大小和 SHA1 校验；
     - delta 应用阶段对每个带 `.shasum` 的 EXE/DLL 做严格的大小 + SHA1 校验，避免产生损坏可执行文件；
     - 构建完成后还会检查关键 EXE 是否存在，防止生成不完整的新目录；
   - 原子切换与回滚：
     - 更新前将原安装目录完整备份，更新成功后再删除备份；
     - 任何一步失败，都尽量通过备份目录恢复到更新前的状态，避免“半新半旧”的安装目录。
