# Bug 修复记录

## 1. 基本信息
- Bug ID: `15284`
- 禅道链接: `https://zentao.creality.com/zentao/bug-view-15284.html`
- 标题: `多盘的时候，冲刷时间 UI 缺失`
- 首次处理日期: `2026-03-19`
- 文档更新日期: `2026-03-19`
- 所属产品: `Sanity Print`
- 所属模块: `切片预览`
- Bug 类型: `代码错误`

## 2. 问题现象
- 多盘预览场景下，左侧预览统计中存在冲刷耗材，但 `Flushed` 对应的时间信息缺失或显示异常。
- 用户侧感知为：冲刷长度/重量存在，但时间口径不完整，无法和总时间对应核对。
- 在多盘场景下，这个问题更明显，因为冲刷时间不是直接统计值，而是 UI 侧通过残差方式反推。

## 3. 禅道重现信息
- `[前置条件]` 使用存在冲刷路径的多盘切片场景。
- `[步骤]` 切片多盘模型，进入预览页面，查看冲刷相关时间显示。
- `[结果]` 冲刷时间 UI 缺失。
- `[期望]` 冲刷时间正常显示，并与真实冲刷段一致。

## 4. 影响范围
- 模块: `G-code 预览统计`
- 关键文件:
  - `src/libslic3r/GCode/GCodeProcessor.hpp`
  - `src/libslic3r/GCode/GCodeProcessor.cpp`
  - `src/slic3r/GUI/GCodeViewer.cpp`
- 受影响流程:
  - 预览页 `Line Type` 的冲刷时间显示
  - 多盘场景下冲刷时间核对

## 5. 根因分析
- 早期实现中，预览页面没有独立维护冲刷时间统计。
- `Flushed` 的时间采用“总时间 - 角色时间 - Move 时间”的方式反推，这种做法在多盘场景下不稳定。
- 冲刷段内的额外同步等待时间没有被明确归属到冲刷时间，导致 UI 侧无法可靠展示真实值。

## 6. 修复策略
- 在底层时间统计中新增独立 `flush_time` 字段，直接记录冲刷时间。
- 对 `FLUSH_START` 到 `FLUSH_END` 之间的真实冲刷段单独累计时间。
- 将冲刷阶段内的额外同步等待时间一并计入冲刷时间。
- 预览层改为直接读取 `time_mode.flush_time`，不再通过残差方式反推。

## 7. 代码改动摘要
- 文件: `src/libslic3r/GCode/GCodeProcessor.hpp`
  - 在 `PrintEstimatedStatistics::Mode` 中增加 `flush_time`。
  - 在时间块标记中增加 `flush_stage`，用于标识冲刷阶段。
- 文件: `src/libslic3r/GCode/GCodeProcessor.cpp`
  - 在时间机计算流程中，独立累计冲刷段时间。
  - 将冲刷阶段的额外同步时间并入冲刷统计。
  - 在最终预估时间统计中回填 `data.flush_time`。
- 文件: `src/slic3r/GUI/GCodeViewer.cpp`
  - `Flushed` 行时间改为直接读取 `time_mode.flush_time`。
  - 百分比优先按冲刷时间占总时间计算，避免 UI 缺失或口径不一致。

## 9. 风险与回滚
- 风险等级: `低`
- 主要风险:
  - 冲刷时间口径从 UI 反推改为底层直接统计后，需要确认与历史页面表现差异是否符合产品预期。
  - 不同固件时间模型下，冲刷段同步时间的归属需继续关注边界场景。
- 回滚方案:
  - 回滚 `GCodeProcessor` 中 `flush_time` 独立统计及 `GCodeViewer` 的直接读取逻辑。
