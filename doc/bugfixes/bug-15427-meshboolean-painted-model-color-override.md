## 2. 问题现象
- 在模型已经做过颜色涂色的场景下，进入 Mesh Boolean（布尔）工具后，模型没有按布尔工具中的 A/B 列表颜色显示。
- 用户侧看到的是：未涂色模型会被布尔颜色正常覆盖，但已涂色模型仍保留原本的涂色显示，导致同一布尔场景中的颜色语义不一致。
- 该问题会直接影响布尔分组识别，用户难以通过画面判断当前模型属于哪一组。

## 3. 禅道重现信息
- `[前置条件]` 导入多个模型，并对其中至少一个模型执行颜色涂色操作。
- `[步骤]` 导入多个模型，涂色后进入布尔。
- `[结果]` 涂色的模型，显示了原本的颜色。
- `[期望]` 进入布尔后，应该按照布尔中的颜色显示。

## 4. 影响范围
- 模块:
  - `Mesh Boolean / 布尔工具`
  - `3D 场景模型渲染`
- 关键文件:
  - `src/slic3r/GUI/Gizmos/GLGizmoMeshBoolean.cpp`
  - `src/slic3r/GUI/3DScene.cpp`
- 受影响流程:
  - 已做颜色涂色模型进入布尔工具后的列表分组显示
  - 布尔工具中 A/B 颜色覆盖与模型实际画面一致性

## 5. 根因分析
- 布尔工具进入时，会通过 volume color override 将参与布尔的模型临时改成布尔列表颜色。
- 对普通模型，这套 override 能正常生效；但对已经做过颜色涂色的模型，渲染会进入 `mmu_segmentation_facets` 的彩色分段渲染分支。
- 该分支原实现始终按涂色后的挤出机颜色渲染各分段，只在极少数透明场景下借用了 override 的 alpha，实际上没有真正使用布尔工具设置的 `force_native_color / force_neutral_color`。
- 结果就是：布尔工具已经下发了颜色覆盖，但彩色模型的实际渲染路径把它绕开了，最终仍显示原本的涂色颜色。

## 6. 修复策略
- 保持布尔工具现有的 volume color override 入口不变，不改动分组和 UI 逻辑。
- 在彩色涂色模型的渲染分支中，补齐对 `force_native_color / force_neutral_color` 的支持。
- 当模型处于这两类“强制颜色”状态时，不再继续按挤出机分段颜色渲染，而是统一使用当前 `render_color`，确保布尔覆盖色真正落到画面上。
- 非布尔覆盖场景仍保持原有彩色分段显示逻辑，避免影响正常的颜色涂色预览。

## 7. 代码改动摘要
- 文件: `src/slic3r/GUI/3DScene.cpp`
  - 在 `GLVolume::simple_render(...)` 中，针对 `mmu_segmentation_facets` 的彩色模型渲染路径新增 `use_render_color_override` 判断。
  - 当 `force_native_color` 或 `force_neutral_color` 为真时，彩色模型的每个分段统一使用 `render_color` 渲染。
  - 仅当不存在强制颜色覆盖时，才继续按原有逻辑使用挤出机颜色渲染涂色分段。

## 9. 风险与回滚
- 风险等级: `低`
- 主要风险:
  - 本次修改位于彩色模型的通用渲染分支，需要关注其他依赖 `force_native_color / force_neutral_color` 的 gizmo 场景是否与预期一致。
  - 若后续还有其他基于分段颜色的特殊渲染需求，需要确认其与“强制颜色覆盖”的优先级关系。
- 回滚方案:
  - 回滚 `src/slic3r/GUI/3DScene.cpp` 中 `GLVolume::simple_render(...)` 对彩色模型强制颜色覆盖的处理逻辑。
