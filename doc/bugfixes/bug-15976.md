## 2. 问题现象
- 连接 K2 打印机后，在设备详情页执行操作，切片软件会出现卡死，严重时表现为卡死崩溃。
- 从禅道标题看，问题是在“耗材余量”相关场景下暴露出来；但结合本次代码改动，实际修复点位于设备详情页的 WebView 路由处理链路。
- 影响: 设备详情页无法继续交互，用户会认为软件失去响应。

## 3. 影响范围
- 模块: `设备管理 / 设备详情页 / WebView 路由`
- 关键文件:
  - `src/slic3r/GUI/print_manage/Routes/DeviceMgrRoutes.cpp`
- 影响流程:
  - 设备详情页触发 `get_webrtc_local_param`
  - HTTPS / `videoEncryption` 视频参数交换链路

## 4. 复现步骤（禅道截图原文整理）
1. 前置条件: 连接 K2 打印机，固件版本 `1.1.3.8.5`，IP 为 `172.23.7.11`。
2. 在设备详情页执行任何操作。
3. 查看切片软件状态。
4. 实际结果: 软件卡死崩溃。
5. 期望结果: 软件正常运行。


## 5. 根因分析
- 设备详情页的 `get_webrtc_local_param` 路由运行在 WebView 回调链路中。
- 修复前，当 `videoEncryption=true`，或 URL 以 `https://` 开头时，代码会在该回调内直接执行同步网络请求:
  - `Http::post(url) ... .perform_sync()`
- 这意味着页面线程会一直等待远端视频服务返回结果，期间 UI 事件无法及时继续处理；一旦网络握手、证书校验或服务端响应变慢，用户侧就会感知为“软件卡死”。
- 同步请求结束后才继续组装 `out_data` 并回传前端，因此成功和失败路径都会阻塞当前页面交互。
- 从本次提交看，问题根因并不在“耗材余量”计算本身，而是在设备详情页场景下伴随触发的 WebRTC 本地参数获取逻辑使用了同步阻塞调用。

注: 本节为根据提交 `c0dd05395` 的代码修改推断出的技术根因，不是禅道页面原文。

## 6. 修复方案
- 将设备详情页 `get_webrtc_local_param` 的结果回传逻辑提取为统一的 `post_result` lambda。
- 加密视频分支不再使用同步 `perform_sync()`，改为异步 `perform()`:
  - 成功回调中组装 `sdp/url/videoEncryption/status`
  - 失败回调中组装 `status/error`，必要时保留返回体
  - 异常路径也统一回传错误信息
- 通过 `wxGetApp().CallAfter(...)` 异步把结果投递回前端，避免在当前页面回调中等待网络请求完成。
- 非加密分支也统一通过 `post_result` 回传，保持前后端消息格式一致。

## 7. 代码改动摘要
- 文件: `src/slic3r/GUI/print_manage/Routes/DeviceMgrRoutes.cpp`
  - 新增 `post_result`，统一封装 `window.handleStudioCmd(...)` 的回传逻辑。
  - 将加密视频分支的请求执行方式从 `.perform_sync()` 改为 `.perform()`。
  - 在 `on_complete` / `on_error` / `catch` 中分别构造回传数据，保证异步场景下状态完整。
  - 非加密分支改为复用同一回传逻辑，减少重复代码。

## 9. 风险与回退
- 风险等级: `中低`
- 主要风险:
  - 请求从同步改为异步后，前端若隐含依赖“立即返回”的时序，需要关注页面状态切换时的回调到达顺序。
  - 如果页面在请求完成前被关闭或切换设备，需要关注回调投递时的对象有效性。
- 回退方案:
  - 回退 `src/slic3r/GUI/print_manage/Routes/DeviceMgrRoutes.cpp` 中本次异步化改动，恢复到 `c0dd05395` 之前版本。
