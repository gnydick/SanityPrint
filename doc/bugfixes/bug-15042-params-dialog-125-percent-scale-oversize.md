- 所属产品: `Sanity Print`
- 所属模块: `准备页面`
- 所属计划: `CP 7.1.0 Release`

- 严重程度: `一般`

- 关联问题: `13999`

## 2. 问题现象
- 在普通分辨率显示器、Windows 缩放为 `125%` 的场景下，打开机型设置或耗材设置时，弹窗整体显示过大。
- 即使不是 4K 屏，只要缩放比例提高，设置页也可能明显大于当前屏幕的合理显示范围。
- 影响：界面密度失衡，弹窗视觉上偏大，影响设置页正常使用体验。

## 3. 禅道复现信息
1. 使用普通分辨率显示器，Windows 缩放设置为 `125%`。
2. 在准备页面打开机型设置或耗材设置。
3. 观察设置弹窗大小。
4. 实际结果：设置页面显示过大。
5. 期望结果：弹窗尺寸稳定，并保持在合理可视范围内。

## 4. 影响范围
- 模块: `参数弹窗 / 机型与耗材设置`
- 关键文件:
  - `src/slic3r/GUI/ParamsDialog.cpp`
- 相关 DPI 基础设施:
  - `src/slic3r/GUI/GUI_Utils.hpp`
- 受影响流程:
  - 打开机型设置弹窗
  - 打开耗材设置弹窗
  - 在普通屏 `125%` 缩放场景下使用参数设置页面

## 5. 根因分析
- 该问题与 bug `13999` 属于同一类根因。
- 旧版外层弹窗尺寸依赖 `em_unit()` 和 `Fit()`，而不是一套稳定、受限的固定基准尺寸。
- `em_unit()` 会随着 DPI/缩放比增大，因此即使只是 `125%` 缩放，也可能把外层弹窗放大到超出预期。
- 缺少基于当前屏幕工作区的边界约束，导致弹窗尺寸受排版度量驱动，而不是受实际可视区域约束。

## 6. 修复策略
- 复用 bug `13999` 中引入的 `ParamsDialog` 受限尺寸方案。
- 为弹窗定义稳定的 DIP 基准尺寸：`1300 x 650`。
- 在应用目标尺寸前，清理旧的 `size hints` 和 `min size`。
- 按当前显示器工作区对弹窗做尺寸与位置限制，保证普通屏 `125%` 缩放下仍然可用。

## 7. 代码改动摘要
- 文件: `src/slic3r/GUI/ParamsDialog.cpp`
  - 增加显示器工作区获取和窗口边界限制 helper。
  - 增加基于 `1300 x 650` DIP 的默认尺寸换算逻辑。
  - 将单纯依赖 `em_unit()` 的外层尺寸方案改为受限的统一尺寸路径。
  - 在弹窗显示和 DPI 变化时重新应用这套受限尺寸逻辑。

- [ ] `1920 x 1080`、`125%` 缩放下，打开耗材设置弹窗，尺寸处于合理范围。
- [ ] 弹窗始终完整显示在当前显示器工作区内。
- [ ] 多次关闭再打开后，弹窗尺寸稳定，不会越来越大。
- [ ] 与 `13999` 的高 DPI 修复路径保持一致，不出现两套不同逻辑。

## 9. 风险与回滚
- 风险等级: `低到中`
- 主要风险: 若后续设置页内容继续增加，当前基准尺寸可能需要针对小屏重新调优。
- 回滚方案: 回退 `ParamsDialog.cpp` 中这套 helper 驱动的尺寸控制逻辑。

## 10. 备注
- 该问题本质上是 bug `13999` 在较低缩放场景下的另一种复现形式。
- 虽然根因一致，但单独记录仍有价值，因为它对应的是“普通屏 + 125% 缩放”这一更常见的用户场景。
