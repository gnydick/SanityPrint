## 2. 问题现象
- 在 `3840 x 2160` 等高分屏、Windows 缩放为 `175%` 的场景下，打开机型管理或耗材管理时，设置弹窗整体显示过大。
- 在回归场景中，如果将弹窗在不同 DPI/缩放比的显示器之间拖动，弹窗还可能继续变大，甚至超出当前屏幕可视区域。
- 影响：设置界面难以操作，部分内容和控件可能超出可见范围。

## 3. 禅道复现信息
1. 使用 `3840 x 2160` 高分屏，Windows 缩放设置为 `175%`。
2. 在准备页面打开机型管理或耗材管理。
3. 观察设置弹窗大小。
4. 可选：将弹窗拖到另一块 DPI/缩放比不同的屏幕上。
5. 实际结果：弹窗大于预期，部分场景下会溢出屏幕。
6. 期望结果：弹窗尺寸稳定，并始终限制在当前显示器工作区内。

## 4. 影响范围
- 模块: `参数弹窗 / 机型与耗材设置`
- 关键文件:
  - `src/slic3r/GUI/ParamsDialog.cpp`
- 相关 DPI 基础设施:
  - `src/slic3r/GUI/GUI_Utils.hpp`
- 受影响流程:
  - 打开机型管理弹窗
  - 打开耗材管理弹窗
  - 在不同 DPI/缩放比显示器之间拖动弹窗

## 5. 根因分析
- `DPIDialog` 本身已经通过 `on_dpi_changed(...)` 接入 DPI 变化回调，所以问题不是“没有 DPI 事件链”。
- 旧版 `ParamsDialog` 的外层尺寸依赖 `em_unit()`，例如 `130 * em_unit()`、`75 * em_unit()`，会随着当前 DPI 缩放系数一起放大。
- DPI 变化时又叠加使用了 `Fit()`，导致“内容驱动尺寸”和“DPI 驱动尺寸”共同作用，最终把外层弹窗放得比预期更大。
- 弹窗尺寸和位置没有按当前显示器工作区做边界约束，因此一旦在高 DPI 屏上被放大，就可能超出屏幕可视范围。

## 6. 修复策略
- 为参数弹窗定义稳定的 DIP 基准尺寸：`1300 x 650`。
- 在设置目标尺寸前，清理旧的 `size hints` 和 `min size`，避免上一块屏幕遗留的尺寸约束继续生效。
- 按当前显示器的工作区对弹窗做边界限制，上限控制在工作区的 `90%` 以内。
- 在构造、弹出和 DPI 变化三个阶段统一复用这一套受限尺寸逻辑。

## 7. 代码改动摘要
- 文件: `src/slic3r/GUI/ParamsDialog.cpp`
  - 新增 `get_display_client_area(...)`，获取弹窗当前所在显示器的工作区。
  - 新增 `get_params_dialog_default_size(...)`，将 `1300 x 650` 的 DIP 基准换算到当前 DPI。
  - 新增 `adjust_dialog_in_screen(...)`，限制弹窗尺寸和位置不超出当前显示器工作区。
  - 新增 `apply_params_dialog_size(...)`，统一清理旧尺寸约束并应用受限默认尺寸。
  - 构造函数不再单纯依赖 `em_unit()` 外层尺寸，而是改为走统一的受限尺寸路径。
  - `Popup()` 中在显示前重新应用一次受限尺寸逻辑。
  - `on_dpi_changed(...)` 改为 `m_panel->msw_rescale(); Layout(); apply_params_dialog_size(...)`，不再使用 `Fit()` 加 `em_unit()` 外层尺寸的组合。

## 9. 风险与回滚
- 风险等级: `中`
- 主要风险: 如果后续设置页内容明显增多，固定 DIP 基准尺寸可能需要重新调优。
- 回滚方案: 回退 `ParamsDialog.cpp` 中这套 helper 驱动的尺寸控制逻辑，恢复旧的 `em_unit()` 外层尺寸方案。
